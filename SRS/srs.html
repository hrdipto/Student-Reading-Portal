1. Introduction

1.1 Purpose

The purpose of this document is to provide a detailed overview of our Reading App software, the initial version of which is 1.0. The document will describe about the targeted audience, it’s user interface, hardware and software requirements. Nonetheless, it will help the designers and developers in the whole software delivery purpose.

1.2 Document Conventions

This document follows MLA Format. Bold-faced text has been used to emphasize section and sub-section headings. Highlighting is to point out words in the glossary and italicized text is used to label and recognize diagrams.

1.3 Intended Audience and Reading Suggestions

This document is intended towards the users, developers, designers, marketing agents, testers and project managers. This document will assess each audience in grasping the total concept about the app and its features.

1.4 Product Scope

This app is based on the purpose of tracking the reading of a student for a specific reading. The users i.e. the students can use this app in order read a given file by a teacher and his reading progress will be tracked by the software and this progress of each student will be then visible to the teacher.

2. Overall Description

2.1 Product Perspective

The product mentioned in the document will be a solution for the teacher in order to track his student’s progress of reading of a file provided by him. The students in general tend to skip the reading materials that the teachers provide which may eventually harm their academic results. So this app will make the students bound to read their materials as the teacher will be able to track their reading progress.

2.2 Product Functions

The major functions of the product are –

  • Let the teacher create a group for students to join.
  • Let the students join a group created by a teacher via access key.
  • Let the teacher assign reading materials to the students of a specific group.
  • Reading progress of each student will be tracked.
  • Eye will be tracked of each student while they read in order as a proof of reading.
  • Show the teacher the reading progress of each student.

2.3 User Classes and Characteristics

2.3.1 Teacher:

Teacher will mostly use this app to assign study materials to his desired number of students via uploading a file to a specific group containing students. And he will be able to track the reading progress of each student via the application.

2.3.2 Student:

Students will mostly use this app to read the study material provided by the teacher of a specific group and then their reading progress will be tracked by the software and will be available for the teacher of the group to see the progress.

2.3.3 Database Administrator:

The Database Administrator is expected to have a college degree and an experience of atleast 2 years in DBA and must have a profound knowledge about IT field. They have the ability to update the informations in the database and manage it accordingly.

2.3.4 Developer:

The developers are expected to have high knowledge about software developing and designing and must have a college degree along with 2 years of software developing experience and also a good knowledge of software related to the product scope. They have to privilege to update and add new features of the app for betterment of user interface.

2.4 Operating Environment

The software will have two platforms in which it will work, one is the website platform in which the teacher will be able to upload files for the students to read and the other is the android app platform in which the student will read the reading materials. In website platform it will be running with the combination of JavaScript, Python and PostgreSql. And on the android platform it will be running with the use of Kotlin.

2.5 Design and Implementation Constraints

  • Platform requirements: The software must work on android and on website.
  • Time constraint: The software must show updates in every 10 seconds.
  • Design constraint: The software must have eye recognition system by which the student’s eye can be tracked in order of a proof of reading.

2.6 User Documentation

For the user documentation please refer to section 3: External Interface Requirements and attached user manual.

2.7 Assumptions and Dependencies

It is assumed that the software will work fine as long it is used on an android device or on a web browser.
As the software is dependent on feeding latest reading progress from the student’s android device through online thus the student is expected to have an internet connection while using the software.

3. External Interface Requirements

3.1 User Interfaces

The user interface of Awesome Reading App has been designed in such a way so that the user doesn’t have to roam around to find options and get annoyed. The interface is easy to use and thus the user will get the hold of the software within seconds.

The home screen will have the option for the user to view or create classes for the students.

After selecting the view classes button and choosing the correct class then the student may read the file in that class in case of the student side and the faculty will be able to monitor his students' progress in the teahcer side.

3.2 Hardware Interfaces

The software will have the capability to run on computers and as well as android device. In computer the software will run as a website application which will be accessible with a browser and on the android device the software will run as android app as well as a web application which will be again accessible by a mobile browser.

3.3 Software Interfaces

The software runs on Java based android app as well as JavaScript based web app. In both cases the software has the ability to recognize the user eye in terms of monitoring the students reading progress.

3.4 Communications Interfaces

Users need to connect the software to the main system by the means of internet. From there they can easily perform the reading app tasks. They will be able to login to the app with their registered email address and password. Whenever the user connects the software to the internet the software will automatically take the required updates. The update will completed within a very short period of time.

4. System Features

4.1 Use case name – Daily Reading Goal

  1. 4.1.1 Objective – The purpose of this case is to take departure point from the user through the means of voice recognition.
  2. 4.1.2 Source – The source of this case is the user.
  3. 4.1.3 Actors – The end user of the device which is the customer would be the actor of this case.
  4. 4.1.4 Flow of events –
  5. 4.1.4.1 Basic Flow –
  6. 4.1.4.1.1 User says the departure point.
  7. 4.1.4.1.2 Software recognizes the point and goes to next feature.
  8. 4.1.4.2 Alternative Flow – At step 4.1.4.1.1, the user decides to type the departure point.
  9. 4.1.4.2.1 User presses cancel button.
  10. 4.1.4.2.2 Software shows text field to enter point.
  11. 4.1.4.2.4 Software takes the chosen point and goes to next feature.
  12. 4.1.4.3 Exception Flow – At step 4.1.4.2, the device can’t recognize the point.
  13. 4.1.4.3.1 – The software shows an error message and let’s the user to again say the departure point.
  14. 4.1.5 Post conditions – The user is taken to the destination selection feature.

4.2 Use case name – Eye Tracking

  1. 4.2.1 Objective – The purpose of this case is to take destination point from the user through the means of voice recognition.
  2. 4.2.2 Source – The source of this case is the user.
  3. 4.2.3 Actors – The end user of the device which is the customer would be the actor of this case.
  4. 4.2.4 Flow of events –
  5. 4.2.4.1 Basic Flow –
  6. 4.2.4.1.1 User says the destination point.
  7. 4.2.4.1.2 Software recognizes the point and goes to next feature.
  8. 4.2.4.2 Alternative Flow – At step 4.2.4.1.1, the user decides to type the destination point.
  9. 4.2.4.2.1 User presses cancel button.
  10. 4.2.4.2.2 Software shows text field to enter point.
  11. 4.2.4.2.3 User types point and presses enter.
  12. 4.2.4.2.4 Software takes the chosen point and goes to next feature.
  13. 4.2.4.3 Alternative Flow – At step 4.2.4.1.1, the user decides to go back and give the departure point again.
  14. 4.2.4.3.1 User presses back button.
  15. 4.2.4.3.2 Software takes the user back to the departure selection menu.
  16. 4.2.4.4 Exception Flow – At step 4.2.4.2, the device can’t recognize the point.
  17. 4.2.4.3.1 – The software shows an error message and let’s the user to again say the destination point.
  18. 4.2.5 Post conditions – The user is taken to the select transportation option feature.

4.3 Use case name – Notify the Instructor

  1. 4.3.1 Objective – The purpose of this case is to take transportation option from the user through the means of selections from a set of available transportations.
  2. 4.3.2 Source – The source of this case is the user.
  3. 4.3.3 Actors – The end user of the device which is the customer would be the actor of this case.
  4. 4.3.4 Flow of events –
  5. 4.3.4.1 Basic Flow –
  6. 4.3.4.1.1 User selects the transportation option.
  7. 4.3.4.1.2 Software takes the option and shows the estimated time and cost for that selected transportation option.
  8. 4.3.4.2 Alternative Flow – At step 4.3.4.1.2, the user decides to view a different transportation option
  9. 4.3.4.2.1 User presses another transportation option button.
  10. 4.3.4.2.2 Software goes back to step 4.3.4.1.2
  11. 4.3.4.3 Exception Flow – At step 4.3.4.1.2, the user decides to go back to home.
  12. 4.3.4.3.1 – The user presses home button.
  13. 4.3.4.3.2 – The user is taken back to take departure point menu.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

The software must work seamlessly while its operation. The reading progress must be updated within every 10 seconds and the software must not crash during the operations. The crash rate must be less than 2 percent.

5.2 Software Quality Attributes

The eye recognition system used by the app must recognize the eye precisely and the success rate must be at least 98 percent. The app must renew it’s database time to time so that the information fetch to the users are accurate and reliable. The user interface must be such that even the person with the lowest technical knowledge can use the app with least difficulties.

For information regarding functional requirements, refer to section 4: System Features.
